# 项目需求、限制、目标与二阶段规划（2026-03-18）

## 文档目的

- 对当前项目现状做统一说明，避免目标漂移。
- 明确本项目在本机环境下的能力边界与约束。
- 给出下一阶段可执行计划，核心目标是扩展到更高性能编码模型。

## 项目范围（当前）

- 项目：`multi-model-chat-demo`（前后端分离）。
- 前端：Vue 3 + Vite。
- 后端：Spring Boot（已支持 OpenAI 兼容接口 `/v1/models`、`/v1/chat/completions`）。
- 数据：MySQL（Docker 容器 `personal-demo-mysql`）。
- 本地模型：Ollama 统一承载，后端通过 `model_config` 动态路由。

## 已完成现状（基线）

1. 后端 OpenAI 兼容接口已落地并可调用。
2. 后端安全过滤链启动问题已修复（`SecurityConfig` 过滤器顺序）。
3. 编码模型计划清单（V3）已入库。
4. DeepSeek-Coder 可用兜底方案已落地（V4）：
   - `deepseek-coder:1.3b-instruct` 已启用。
   - `/v1/models` 可见。
   - `/v1/chat/completions` 可正常返回。
5. 当前本地已拉取模型：
   - `deepseek-coder:1.3b-instruct`
   - `gemma2:2b`

## 核心需求（统一口径）

### 功能需求

- 本地后端需持续对外提供 OpenAI 兼容 API。
- 支持多模型配置、启停、切换，且不改代码即可扩展新模型。
- 编码场景优先，优先保障代码问答、代码生成与修复任务。

### 非功能需求

- 可观测：可判断模型是否在线、是否可被路由、响应是否成功。
- 可运维：模型计划、已部署清单、失败原因需文档化。
- 可演进：后续可平滑迁移到更高算力机器或云端 GPU。

## 当前限制（必须承认）

1. 本机 GPU：`NVIDIA GeForce GTX 1650 4GB`，不适合稳定承载 `>7B` 编程模型在线服务。
2. Ollama 模型标签可用性不稳定，部分标签可能 `manifest not found`。
3. 本地网络对 Docker Hub/Maven 存在波动，镜像与依赖拉取偶发失败。
4. MVP 阶段 `api_key_encrypted` 仍是明文写入逻辑，生产不可直接沿用。

## 二阶段目标（主目标）

- 从“小模型可用”升级到“更高性能编码模型可用”。
- 目标模型优先级：
  1. `deepseek-coder-v2-lite-instruct:7b`
  2. `qwen2.5-coder:7b-instruct`

## 二阶段执行规划

### 第 1 步：模型可用性确认与命名对齐

- 对目标模型进行 Ollama 真实标签探测（避免仅按猜测标签写配置）。
- 若官方标签与计划名不一致，以“实际可拉取标签”为准落库。

### 第 2 步：部署策略分流（按算力）

- 本机策略（短期）：保留 `1.3b/2b` 作为默认服务与兜底。
- 增强策略（推荐）：将 `7b` 部署到高显存主机（>=16GB VRAM）或云 GPU。
- 路由策略：同一 OpenAI 兼容入口下按模型编码透明切换。

### 第 3 步：接入与验收

- 在 `model_config` 中新增并启用目标 `7b` 模型。
- 验收口径：
  - `/v1/models` 可见目标模型；
  - `/v1/chat/completions` 调用成功；
  - 与 `1.3b` 对比延迟与结果质量可记录。

### 第 4 步：性能基线与容量规划

- 固定测试集评估：首 token 延迟、总时延、稳定性（连续调用）。
- 输出扩容建议：并发阈值、超时阈值、推荐硬件档位。

## 里程碑定义

- M1：本地链路稳定（已完成）。
- M2：`deepseek-coder-v2-lite-instruct:7b` 可通过统一 API 调用。
- M3：`qwen2.5-coder:7b-instruct` 可通过统一 API 调用。
- M4：形成“本地 + 高算力节点”双层模型服务方案。

## 结论

- 当前项目已完成“本地可用”阶段，具备继续扩展的工程基础。
- 下一步工作重点已明确：围绕 `deepseek-coder-v2-lite-instruct:7b` 与 `qwen2.5-coder:7b-instruct` 完成可用化与性能升级。
